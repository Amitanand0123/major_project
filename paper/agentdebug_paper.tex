\begin{abstract}
Large Language Model (LLM) agents are increasingly capable at software engineering, yet we still understand surprisingly little about \emph{how} they fail. Current analysis methods rely on expensive human annotation or commercial API calls, examine errors through a single detection lens, and lack systematic ways to distinguish root causes from downstream symptoms. We present AgentDebug, a framework for detecting, diagnosing, and categorizing errors in LLM coding-agent trajectories. The core contribution is a \emph{dual-channel detection} pipeline that runs two independent channels---95 hand-crafted regex patterns and a locally hosted LLM---over every step of an agent trajectory, followed by \emph{counterfactual root-cause identification} that pinpoints the single most consequential error per trajectory. On 300 SWE-bench trajectories, dual-channel analysis reveals a stark asymmetry: the LLM channel detects 8.2$\times$ more errors than regex (13,616 vs.\ 1,662), with the two channels agreeing on module-level error state 69.1\% of the time. Regex systematically over-represents Action and System errors (62.4\%) while under-detecting cognitive failures (Memory, Reflection, Planning), which the LLM channel captures at 58.7\%. A frequency--criticality inversion emerges: rare System errors cause 2.0$\times$ more root-cause failures than their frequency predicts, while common Action errors are less likely to be the actual root cause. These findings provide actionable guidance for agent builders: syntax pre-validation and scope-aware editing would address over 60\% of detected errors. All inference runs locally through Ollama with Qwen2.5-Coder:7B on a consumer-grade Windows machine at zero API cost. Code and data are available at \url{https://github.com/Amitanand0123/major_project}.
\end{abstract}

\section{Introduction}
\label{sec:introduction}

\PARstart{L}{arge} Language Model agents are increasingly tackling real-world software engineering problems. Benchmarks like SWE-bench \cite{b1} provide a controlled setting for evaluating how well these agents resolve genuine GitHub issues, and frameworks such as ReAct \cite{b2}, SWE-agent \cite{b3}, and AutoCodeRover \cite{b4} keep raising the bar for automated coding assistants. Most of this progress, however, has focused on making agents succeed more often. Far less effort has gone into understanding the specific ways they break down along the way.

The analysis tools that do exist share two blind spots. First, they are expensive to run: the AgentErrorBench study required ten expert annotators for 200 trajectories \cite{b5}, and API-based detection with GPT-4 costs roughly \$150--\$200 per thousand trajectories---manageable for well-funded labs, but a real barrier for smaller teams. Second, every existing approach relies on a single detection method, making it impossible to know what that method is missing.

AgentDebug addresses both issues. It runs two detection channels side by side over every step of an agent trajectory: a deterministic regex channel built from 95 patterns that match syntactic error signatures, and a semantic channel powered by a locally hosted LLM (Qwen2.5-Coder:7B via Ollama). Because the two channels work independently, we can directly compare their outputs and quantify exactly where they agree and disagree.

\subsection{Research Questions}

Our investigation is organized around four questions:

\begin{itemize}
    \item \textbf{RQ1:} What types and frequencies of errors do LLM agents exhibit on software engineering tasks, and how are those errors spread across cognitive modules?
    \item \textbf{RQ2:} To what extent do pattern-based and LLM-based detection agree, and where do their blind spots lie?
    \item \textbf{RQ3:} Which errors actually cause trajectories to fail, and does an error being common make it more likely to be the root cause?
    \item \textbf{RQ4:} What concrete changes to agent architectures would reduce the failure patterns we observe?
\end{itemize}

\subsection{Contributions}

\begin{enumerate}
    \item \textbf{Dual-channel detection.} We are, to our knowledge, the first to apply two independent detection methods to every step of an LLM agent trajectory. On 300 SWE-bench trajectories the LLM channel finds 8.2$\times$ more errors than the regex channel, and the two channels agree on module-level error state 69.1\% of the time.

    \item \textbf{Code-domain error taxonomy.} We extend the AgentErrorTaxonomy of Zhu~et~al.\ \cite{b5} with 20 error types grouped into 5 cognitive modules (19 predefined plus one emergent type, \texttt{wrong\_file\_edit}, surfaced by the LLM channel), providing the first structured vocabulary designed specifically for code-agent failure analysis.

    \item \textbf{Module-level asymmetry.} We show that regex patterns systematically over-represent Action and System errors (62.4\% combined) while under-representing cognitive errors (37.6\% vs.\ 58.7\% on the LLM side), an asymmetry that could skew the conclusions of any study relying on patterns alone.

    \item \textbf{Frequency--criticality inversion.} We demonstrate that error frequency is a poor proxy for error importance: rare System errors are 2.0$\times$ more likely to be root causes than their frequency predicts, while common Action errors are disproportionately less critical.

    \item \textbf{Zero-cost, consumer-hardware pipeline.} The entire analysis runs on a Windows~11 laptop with 16\,GB RAM at \$0.00 in API fees, making it straightforward for any group to replicate.
\end{enumerate}

\section{Related Work}
\label{sec:related}

\subsection{LLM Agents and Benchmarks}

SWE-bench \cite{b1} and SWE-bench+ \cite{b16} set the standard for evaluating coding agents on real GitHub issues. A steady stream of agent architectures---ReAct \cite{b2}, SWE-agent \cite{b3}, AutoCodeRover \cite{b4}, OpenHands \cite{b13}, CodeAct \cite{b14}---has pushed resolution rates higher, while broader benchmarks like AgentBench \cite{b23} and TheAgentCompany \cite{b24} keep exposing weaknesses in long-horizon reasoning. Nearly all of this work is aimed at making agents succeed more often; ours asks a different question: \emph{why} do they fail?

\subsection{Agent Failure Analysis and Debugging}

On the failure-analysis side, Zhu~et~al.\ \cite{b5} introduced the AgentErrorTaxonomy and the original AgentDebug framework, relying on GPT-4 for single-channel detection and simulated self-assessment. Cemri~et~al.\ \cite{b6} proposed MAST for multi-agent failures; Winston~\cite{b7} offered a taxonomy for tool-augmented LLMs; Islam~et~al.\ \cite{b15} analyzed 1,187 bug reports across agent frameworks. In the code-debugging literature, Li~et~al.\ \cite{b19} combined LLMs with static analysis, Xia~et~al.\ \cite{b21} and Zhang~et~al.\ \cite{b20} surveyed LLM-based program repair, Tambon~et~al.\ \cite{b25} catalogued bug patterns in LLM-generated code, and Chen~et~al.\ \cite{b8} explored self-debugging. Xia~et~al.\ \cite{b18} showed that some tasks can be solved even without agentic scaffolding. A recurring concern with LLM-based detection is hallucination \cite{b17, b22}; our LLM channel flags roughly 8$\times$ more errors than regex, a non-trivial fraction of which are likely false positives (Section~\ref{sec:discussion}). We use counterfactual reasoning \cite{b9} in Phase~2 to separate root causes from downstream symptoms.

Our work extends this landscape in four ways: (1)~code-domain-specific error types; (2)~dual-channel parallel detection; (3)~fully local inference at zero API cost; and (4)~quantified module-level asymmetry between channels.

\section{Methodology}
\label{sec:methodology}

\subsection{Problem Formulation}

Let $T = \{s_1, s_2, \ldots, s_n\}$ be an agent trajectory of $n$ steps, where each step $s_i$ pairs an observation $o_i$ (environment feedback) with an action $a_i$ (agent output). Our goals are:

\begin{enumerate}
    \item \textbf{Detect} errors through two independent channels, a regex channel $E^R$ and an LLM channel $E^L$, characterizing each error by its cognitive module, taxonomy type, and step location.
    \item \textbf{Compare} the channels by computing, for every step--module pair, whether both detected an error, neither did, or only one of them did.
    \item \textbf{Identify} the root-cause error $e^* \!=\! \arg\max_{e \in E^R \cup E^L}\text{impact}(e)$ via counterfactual reasoning.
\end{enumerate}

Fig.~\ref{fig:architecture} shows the complete pipeline.

\centering

Phase~2 applies counterfactual reasoning with three-layer validation to identify the root-cause error. Phase~3 generates corrective patches and verifies them against real test suites.}

\subsection{Phase 1: Dual-Channel Fine-Grained Detection}

Both channels process every step independently; neither has access to the other's output.

\subsubsection{Channel A: Pattern-Based (Regex) Detection}

The regex channel uses a hand-curated library of 95 regular expressions, each targeting a specific syntactic footprint. Some fire on \texttt{Indentation\-Error} tracebacks, others on \texttt{Module\-Not\-Found\-Error} messages, and a handful catch repeated failed commands. At every step we run every pattern against both the observation and the action text. A match is recorded with confidence~1.0, making the channel fully deterministic and reproducible. \subsubsection{Channel B: LLM-Based Semantic Detection}

The second channel sends each step---its observation, action, and three preceding steps for context---to Qwen2.5-Coder:7B running locally via Ollama. The model classifies errors across all five modules simultaneously. Unlike regex, it reads the meaning of the code rather than scanning for string patterns, so it can flag problems that leave no syntactic trace: editing the wrong file, drifting outside the intended scope, or misreading a test result.

\subsubsection{Agreement Computation}

For each step $s_i$ and module $m \in \mathcal{M}$ (where $\mathcal{M}$ = \{memory, reflection, planning, action, system\}), let $R = E^R(s_i, m)$ and $L = E^L(s_i, m)$ denote the regex and LLM error sets respectively. The agreement label is:

\begin{equation}
\text{agree}(s_i, m) \!=\! \begin{cases}
\texttt{both\_err} & R \!\neq\! \emptyset \,\wedge\, L \!\neq\! \emptyset \\
\texttt{both\_cln} & R \!=\! \emptyset \,\wedge\, L \!=\! \emptyset \\
\texttt{rgx\_only} & R \!\neq\! \emptyset \,\wedge\, L \!=\! \emptyset \\
\texttt{llm\_only} & R \!=\! \emptyset \,\wedge\, L \!\neq\! \emptyset
\end{cases}
\end{equation}

The overall agreement rate is:
\begin{equation}
\text{Agree} = \frac{N_{\textit{both\_err}} + N_{\textit{both\_cln}}}{N_{\textit{total}}}
\end{equation}

\subsubsection{End-to-End Pipeline}

Algorithm~\ref{alg:pipeline} summarizes the full AgentDebug pipeline.

\begin{algorithm}[h!]
\caption{AgentDebug Dual-Channel Pipeline}
\label{alg:pipeline}
\begin{algorithmic}[1]
\REQUIRE Trajectory $T = \{s_1, \ldots, s_n\}$
\ENSURE Root-cause $e^*$, agreement labels
\STATE $\mathcal{M} \leftarrow \{\text{Mem, Ref, Plan, Act, Sys}\}$
\STATE $E^R \leftarrow \emptyset$;\ $E^L \leftarrow \emptyset$
\FOR{each step $s_i \in T$}
    \STATE \COMMENT{Channel A: Regex (deterministic)}
    \FOR{each pattern $p$ in 95-pattern library}
        \IF{$p$ matches $s_i$.\textit{observation} or $s_i$.\textit{action}}
            \STATE $E^R \leftarrow E^R \cup \{(i, \text{module}(p), \text{type}(p))\}$
        \ENDIF
    \ENDFOR
    \STATE \COMMENT{Channel B: LLM (one call per step)}
    \STATE $\textit{resp} \leftarrow \text{LLM}(s_i,\ s_{i-3:i-1})$ \COMMENT{3-step context}
    \FOR{each $m \in \mathcal{M}$}
        \IF{$\textit{resp}$ reports error for $m$}
            \STATE $E^L \leftarrow E^L \cup \{(i, m, \textit{resp}.\text{type}(m))\}$
        \ENDIF
    \ENDFOR
    \STATE \COMMENT{Compute agreement per module}
    \FOR{each $m \in \mathcal{M}$}
        \STATE $\text{agree}(s_i, m) \leftarrow \text{label}(E^R, E^L, i, m)$ \COMMENT{Eq.~1}
    \ENDFOR
\ENDFOR
\STATE \COMMENT{Phase 2: Root-cause (one LLM call)}
\STATE $e^* \!\leftarrow\! \text{LLM\_CF}(E^R \!\cup\! E^L, T)$
\STATE $e^* \!\leftarrow\! \text{validate\_3layer}(e^*)$
\RETURN $e^*$, $\{\text{agree}(s_i, m)\}$
\end{algorithmic}
\end{algorithm}

\subsection{Phase 2: LLM-Powered Critical Error Identification}

Once Phase~1 has flagged errors from both channels, Phase~2 takes the combined set and narrows it down to the single most consequential error---the root cause that, if avoided, would most likely have changed the trajectory's outcome.

\subsubsection{Counterfactual Reasoning}

The local LLM examines each detected error and asks a simple question: ``If this error had not occurred, would the agent likely have finished the task?'' The error that scores highest on this counterfactual test is labeled the root cause.

\needspace{12\baselineskip}
\subsubsection{Three-Layer Validation}

LLM output is not always well-formed, so we enforce taxonomy compliance through three successive layers:

\begin{enumerate}
    \item \textbf{Prompt engineering.} The prompt lists every valid module and type along with formatting rules and worked examples. This alone yields taxonomy-compliant output for roughly 94\% of calls.
    \item \textbf{Post-parsing validation.} A programmatic check confirms the returned module and type exist in the taxonomy; minor deviations (casing, extra whitespace) are corrected automatically, catching about 80\% of violations that slip past Layer~1.
    \item \textbf{Fallback correction.} Any remaining invalid value is mapped to its closest valid entry by string similarity.
\end{enumerate}

After all three layers, 12 of 293 Phase~2 outputs (4.1\%) still carried labels outside the formal taxonomy (\texttt{format\_error}, \texttt{logic\_error}). We keep these as-is rather than force-mapping them, since they describe recognizable error behaviors that our taxonomy simply does not cover yet.

\subsubsection{Local LLM Configuration}

All inference uses Ollama \cite{b11} with Qwen2.5-Coder:7B \cite{b12} (4.7\,GB quantized), temperature set to 0.0 for reproducibility. No API keys are needed and no data leaves the machine.


\subsection{Phase 3: Iterative Debugging with Real Verification (Framework)}
\label{sec:phase3}

The AgentDebug architecture includes a third phase that closes the loop from detection to repair. Given the root-cause error $e^*$ from Phase~2, Phase~3 proceeds in up to $K=3$ iterations: (1)~the LLM generates targeted corrective feedback grounded in the dual-channel evidence and propagation chain, (2)~it produces a corrected code edit as a unified-diff patch, (3)~it predicts whether the correction would lead to task success, and (4)~the feedback is scored on specificity and actionability. The loop terminates early if the model predicts success with high-quality feedback.

The generated patch is then verified against the project's real test suite in an isolated virtual environment: the target repository is cloned at the correct base commit, dependencies are installed, the patch is applied via \texttt{git apply}, and the SWE-bench \texttt{FAIL\_TO\_PASS} tests are run under a 600-second timeout. A \emph{gold-patch baseline} verifies the known-correct SWE-bench patch first, so that environment failures (missing C extensions, Python version mismatches) can be distinguished from genuine patch failures. A four-tier comparison (both pass, gold only, corrective only, neither) then cleanly separates patch quality from infrastructure noise.

This infrastructure is model-agnostic and fully implemented in the codebase. Scaling the evaluation to larger samples with more capable models is a primary direction for future work.

% VIII. ACTIONABLE INSIGHTS
% ============================================================
\section{RQ4: Actionable Insights from Error Analysis}
\label{sec:insights}

The dual-channel data point to four concrete interventions that agent builders could implement today:

\begin{enumerate}
    \item \textbf{Syntax pre-validation.} Running an AST parser and whitespace linter \emph{before} the agent submits code would catch \texttt{syntax\_\allowbreak{}error} (31.5\% of regex detections) and \texttt{indentation\_\allowbreak{}error} (14.8\%)---together nearly half of all pattern-detected errors and the single largest category of root-cause failures.

    \item \textbf{Scope-aware editing.} A lightweight scope guard that restricts edits to task-relevant files would address \texttt{wrong\_\allowbreak{}file\_\allowbreak{}edit} (22.5\% of LLM detections) and \texttt{scope\_\allowbreak{}violation} (17.1\%), which together make up close to 40\% of the LLM channel's output.

    \item \textbf{Persistent context memory.} Maintaining a running log of imports, visited files, and prior modifications would mitigate three related memory failures: \texttt{dependency\_\allowbreak{}omission} (17.9\%), \texttt{retrieval\_\allowbreak{}failure} (5.6\%), and \texttt{file\_\allowbreak{}location\_\allowbreak{}forgetting} (3.7\%).

    \item \textbf{Reflection quality gates.} Requiring the agent to verify its own assertions against concrete evidence before moving on would reduce \texttt{progress\_\allowbreak{}misjudge} (10.3\%) and \texttt{outcome\_\allowbreak{}misinterpretation} (5.0\%).
\end{enumerate}


% ============================================================
% IX. DISCUSSION AND LIMITATIONS
% ============================================================
\section{Limitations}
\label{sec:discussion}

Several limitations apply. The taxonomy and regex patterns target Python-based SWE-bench trajectories from a single agent framework (SWE-agent with LLaMA); error distributions may differ for other languages, domains, or architectures. Both the Phase~1 LLM channel and the Phase~2 root-cause identifier use the same 7B model class (Qwen2.5-Coder:7B), creating a risk of systematic blind spots and precluding inter-rater reliability assessment. The LLM channel's 8.2$\times$ detection multiplier includes interpretive judgments (e.g., ``scope violation,'' ``wrong file edit'') that have not been validated against human ground truth; a preliminary inspection of 100 LLM-detected errors suggests 60--80\% are plausible, but a rigorous precision study with multiple annotators remains future work. Finally, the term ``error'' carries different meanings across the two channels---regex flags syntactic events while the LLM labels semantic failures---so the 8.2$\times$ multiplier reflects a broader definition of error rather than 8.2$\times$ more ``real'' mistakes.


% ============================================================
% X. CONCLUSION AND FUTURE WORK
% ============================================================
\section{Conclusion and Future Work}
\label{sec:conclusion}

AgentDebug is a framework for detecting, diagnosing, and categorizing errors in LLM coding-agent trajectories. Its core contribution is dual-channel detection: on 300 SWE-bench trajectories, the LLM channel detects 8.2$\times$ more errors than the regex channel, the two channels agree on module-level error state 69.1\% of the time, and regex systematically over-represents Action errors while under-detecting cognitive failures. A frequency--criticality inversion reveals that rare System errors cause 2.0$\times$ more root-cause failures than their frequency predicts, while common Action errors are less likely to be the actual root cause. These findings provide the first quantitative evidence that single-channel detection---whether pattern-based or LLM-based---gives an incomplete picture of agent failures, and that error frequency alone is a poor proxy for error importance. The entire pipeline runs at zero API cost on consumer hardware.

The framework also includes Phase~3 (Section~\ref{sec:phase3}) for iterative debugging with real test-suite verification. Future work includes evaluating Phase~3 on hundreds of trajectories with more capable models, conducting a formal precision study of LLM-only detections with multiple human annotators, and extending detection to additional agent frameworks and programming languages. All code and data are available at \url{https://github.com/Amitanand0123/major_project}.
